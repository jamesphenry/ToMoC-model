# 2026-07-14 — 100ep single-variable (CONCLUSIVE)

## Question
Is get_time's 0% a STRUCTURAL failure of the tiny arch, or just UNDERTRAINING?
Single-variable test: same 2.3M control arch (192/4/6), same data/LR,
only epochs 60 -> 100.

## Run (valid — confound fixed)
- baseline-60ep:         d192/4/6, 2.3M, 60ep, loss 0.096
- baseline-100ep-control: d192/4/6, 2.3M, 100ep, loss 0.0383
- (Prior 100ep @ 4.9M was INVALID — bigger arch + more epochs; discarded.)

## VERDICT
get_time was PURE UNDERTRAINING, not a structural ceiling. The `get_t_t_t`
repetition collapse was a learned-but-incomplete pattern, resolved by more
epochs on the SAME tiny model. under_call = 0.0000 (never misses a needed
tool). over_call 0.347 (emits tool when gold is answer_direct) = remaining slack.

## LESSON (to memory)
For a from-scratch TINY model, EPOCHS >> capacity tweaks. Bumping epochs
60->100 on 2.3M fixed every weak class — no arch change, no "cheating".
Capacity (bigger model) is a LAST-DITCH lever, not a tuning knob. Matches
user's "smallest/fastest model possible" principle. The earlier 4.9M/100ep
run was a confound caused by argparse defaults drifting to 256/6/8; restored
to 192/4/6 control.
